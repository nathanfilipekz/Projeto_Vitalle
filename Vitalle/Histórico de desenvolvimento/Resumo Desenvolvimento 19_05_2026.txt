4. POSSÍVEIS BUGS E PROBLEMAS DE QUALIDADE
============================================================

CRÍTICOS (segurança / correção):
- payments.service.processWebhook: NÃO valida assinatura
  do PagBank. Qualquer um pode chamar POST /payments/webhook/
  pagbank e marcar pagamentos como APPROVED -> ativar
  assinatura. Bug de segurança grave.
- whatsapp.controller.handleWebhook: NÃO valida assinatura
  X-Hub-Signature-256 da Meta. Risco de spoofing.
- whatsapp.service.processInboundMessage e
  confirmNextAppointment: a busca por patient.findFirst usa
  apenas {phone contains ...} sem filtrar tenantId, ou seja,
  se dois tenants tiverem pacientes com o mesmo telefone
  pode confirmar a consulta do tenant errado. Vazamento
  cross-tenant.
- auth.service.register: o findFirst de usuário existente
  usa apenas {email}, mas o unique do schema é
  (email, tenantId). Isso impede dois tenants de
  compartilharem e-mail (regra de negócio inconsistente -
  precisa decidir: e-mail global ou por tenant).
- auth.controller.logout: rota declarada como POST /logout
  mas NÃO está sob JwtAuthGuard. Qualquer um com o
  refreshToken consegue chamar (não é fim do mundo porque
  precisa do token, mas o swagger e a doc dizem "Yes" para
  Auth - inconsistência).
- AuditInterceptor não é registrado globalmente; a maioria
  das mutations não está sendo auditada (apesar do
  Claude.md.txt dizer "Sempre criar logs").
- GlobalExceptionFilter não é aplicado em main.ts.
- ThrottlerModule sem APP_GUARD: rate-limit configurado
  mas inefetivo. Login fica vulnerável a brute-force.
- Sem RLS aplicado (não há migration), portanto a
  "double protection" da arquitetura hoje é só camada de
  aplicação.

ALTOS (consistência / lógica):
- appointments.service.create: o conflict check só compara
  dateTime exato. Não detecta sobreposição (ex.: consulta
  de 60 min iniciando às 09:00 conflita com slot 09:30,
  mas o sistema permite).
- appointments.service.cancel: recebe "reason" mas NÃO
  persiste. Apenas delega para updateStatus(... 'CANCELED').
  O campo cancelReason no schema nunca é preenchido.
- patients DTO valida CPF apenas por MinLength(11), sem
  formato ou dígito verificador.
- date strings (dateOfBirth, etc.) vêm como string sem
  IsDateString - validação fraca.
- JwtStrategy não checa a versão/expiração do tenant
  (tenant pode estar com deletedAt mas o token continua
  válido até 15min).
- subscriptions.service.acceptContract existe (legado) e o
  ContractsService.acceptContract também ativa a
  subscription. Há lógica duplicada e potencialmente
  divergente.
- Anamnesis no schema: NÃO existe modelo separado. UI
  do front trata anamnese como entidade independente, mas
  backend mistura com MedicalRecord.

MÉDIOS (qualidade / DX):
- DashboardLayout do front faz redirect a /login com base
  em isAuthenticated vindo do persist, mas a flag isLoading
  começa em true e nunca volta a false em fluxo real
  (só setUser zera). Pode haver flicker / loop.
- Login do front gera um usuário fake e empurra para
  /dashboard sem token - quebra qualquer chamada futura à API.
- next.config.mjs sem `output: 'standalone'` quebra o
  Dockerfile do frontend (multi-stage espera

- Falta arquivo prisma/seed.ts referenciado em package.json.
- Sem testes (jest configurado, mas sem nenhum .spec.ts).
- README.md inutil.

BAIXOS (cosméticos / pequenos):

5. PONTOS CRÍTICOS DE ATENÇÃO
============================================================

a) Multi-tenant: o discriminator está aplicado nas tabelas,
   mas faltam aplicar RLS e revisar TODOS os finds para
   garantir o filtro de tenantId. O caso do WhatsApp já
   prova que dá pra escapar do isolamento.

b) Webhooks públicos: PagBank e WhatsApp precisam validar
   assinatura antes de mutar dados. Hoje é exploit aberto.

c) Migrações Prisma: o repositório não tem nenhuma
   migration, então qualquer deploy depende de
   `prisma migrate dev` na hora - inaceitável em produção.
   Antes de qualquer deploy real, gerar a migration inicial
   e versionar.

d) Frontend desacoplado: a UI está pronta mas vive de mocks.
   Sem integração, o produto não existe. A camada
   services/ e hooks/ precisa ser criada do zero.

e) Fluxo de onboarding (register -> checkout -> aceite
   contrato -> liberar app): existe no backend de forma
   parcial mas o frontend NÃO conduz o usuário por ele.
   Hoje, após register, o app jogaria o usuário no
   dashboard sem ter pago.

f) Compliance / LGPD: dados sensíveis (anamnese, prontuário,
   CPF) não são criptografados em repouso na aplicação.
   Confiar só no PostgreSQL TLS pode não ser suficiente.

g) Build do container do frontend está quebrado pela
   ausência de `output: 'standalone'` no next.config.mjs.
