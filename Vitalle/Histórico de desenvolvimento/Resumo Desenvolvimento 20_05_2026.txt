7. BLOQUEIO DE AGENDAMENTO SOBREPOSTO
============================================================

Regra: o mesmo profissional (doctor) não pode ter duas
consultas cujos intervalos [date_time, date_time + duration)
se sobreponham. Consultas em CANCELED, NO_SHOW ou soft-
deleted são ignoradas.

Migration aplicada:
  009_no_overlap_per_doctor
       - CREATE EXTENSION IF NOT EXISTS btree_gist.
       - Função IMMUTABLE public.appt_range(dt, dur):
           SELECT tstzrange(dt, dt + make_interval(mins => dur), '[)')
       - ALTER TABLE public.appointments ADD CONSTRAINT
         appointments_no_overlap_per_doctor EXCLUDE USING gist (
           doctor_id WITH =,
           public.appt_range(date_time, duration) WITH &&
         ) WHERE (deleted_at IS NULL
                  AND status NOT IN ('CANCELED', 'NO_SHOW'));
       - 1 registro duplicado de teste foi removido antes
         da criação da constraint (id terminado em 7e43).
       - Smoke test via DO $$ ... EXCEPTION WHEN
         exclusion_violation $$ confirmou que a inserção
         conflitante é rejeitada.

Frontend:
  frontend/src/services/appointments-service.ts            [editado]
       - hasOverlap(tenantId, doctorId, dateTimeIso,
         duration): consulta candidates com date_time <
         newEnd e verifica end > newStart no client.
       - createAppointment: chama hasOverlap antes do
         insert; em caso de corrida, traduz o erro
         23P01 / "appointments_no_overlap_per_doctor"
         para mensagem amigável em pt-BR.

Backend:
  backend/src/modules/appointments/appointments.service.ts [editado]
       - create() substituiu o findFirst exato por
         $queryRaw usando public.appt_range() para
         detectar overlap; em caso de corrida, traduz
         a exclusion_violation em BadRequestException
         com mensagem em pt-BR.

Mensagem mostrada ao usuário em qualquer dos pontos:

   profissional. Escolha outro horário ou ajuste a duração."

11. O QUE FALTOU / PRÓXIMOS PASSOS
============================================================

Pré-requisitos para o backend NestJS rodar conectado:

  (A) Substituir [YOUR-DB-PASSWORD] em backend/.env pela
      senha do Postgres do projeto Supabase
      (painel: Project Settings > Database).

  (B) Substituir [COLOQUE-A-SERVICE-ROLE-KEY-DO-PAINEL-AQUI]
      por SUPABASE_SERVICE_ROLE_KEY (painel: Project
      Settings > API). Necessária se você quiser bypass
      total de RLS pelo backend.

  (C) Trocar JWT_SECRET e JWT_REFRESH_SECRET por valores
      gerados com `openssl rand -hex 32`.

  (D) Rodar localmente:
        cd backend
        npm install
        npx prisma generate
        npx prisma migrate diff \
          --from-empty \
          --to-schema-datamodel prisma/schema.prisma \
          --script > prisma/migrations/0_init/migration.sql
        npx prisma migrate resolve --applied "0_init"
      Isso versiona a baseline (o banco já tem o schema;
      essa rotina só "marca" a migration como aplicada
      para evitar drift).

  (E) Quando o backend estiver no ar, dropar as policies
      "*_mvp_anon" do banco (migration de reversão):
        DROP POLICY tenants_mvp_anon ON public.tenants;
        DROP POLICY users_mvp_anon ON public.users;
        DROP POLICY doctors_mvp_anon ON public.doctors;
        DROP POLICY patients_mvp_anon ON public.patients;
        DROP POLICY appointments_mvp_anon ON public.appointments;
